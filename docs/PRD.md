# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트 명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고. 관리자가 주문을 관리할 수 있는 간단한    풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴선택이나 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프론트 앤드: HTML, CSSM 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터 베이스: PostgreSQL

## 3. 기본 사항
- 프로트 앤드와 백앤드를 따로 개발
- 기본적인 앱 기술만 사용
- 학습 목적이므로 사용인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 화면 별 PRD

### 4.1 주문하기 화면

#### 1) 목적
- 사용자가 커피 메뉴(ICE/HOT/카페라떼)를 선택하고, 옵션(샷 추가/시럽 추가)을 반영해 장바구니에 담은 뒤 주문을 생성할 수 있도록 한다.
- 사용자는 주문 완료 여부를 즉시 인지할 수 있어야 한다.

#### 2) 사용자
- 일반 사용자(고객): 메뉴 선택 및 주문 생성을 수행한다.
- 관리자는 이 화면에서는 조회 전용 또는 이동만 수행한다. (관리자 화면 PRD는 추후 작성)

#### 3) 핵심 사용자 시나리오
1. 사용자가 상단에서 `주문하기` 탭을 확인한다.
2. 사용자가 메뉴 카드에서 상품과 옵션을 선택한 뒤 `담기`를 클릭한다.
3. 장바구니 패널에서 담긴 항목, 수량, 총 금액이 즉시 갱신된다.
4. 사용자가 `주문하기` 버튼을 클릭하면 주문이 생성되고, 성공 메시지와 함께 장바구니가 비워진다.
5. 네트워크/서버 오류 발생 시 사용자에게 오류가 안내된다.

#### 4) 화면 구성(UI)
##### (1) 공통 상단 영역
- 앱명 `COZY`
- 상단 탭/네비게이션: `주문하기`, `관리자`
- 현재 화면은 `주문하기`가 활성 상태로 표시된다.

##### (2) 메뉴 리스트 영역
- 카드 UI로 메뉴를 노출한다.
- 와이어프레임에 맞춰 최소 3개 상품을 제공한다.
  - 아메리카노(ICE): 4,000원
  - 아메리카노(HOT): 4,000원
  - 카페라떼: 5,000원
- 각 카드에는 다음 요소가 포함된다.
  - 메뉴명
  - 기본 가격
  - 옵션 체크박스(와이어프레임 기준)
    - 샷 추가: `+500원`
    - 시럽 추가: `+0원`
  - CTA 버튼: `담기`

##### (3) 장바구니 영역(하단 패널)
- 장바구니에 담긴 항목을 리스트로 보여준다.
- 각 항목은 다음 정보를 포함한다.
  - 상품명(예: `아메리카노(ICE)`)
  - 선택된 옵션 요약(예: `샷 추가`, `시럽 추가`)
  - 수량(예: `x 1`)
  - 라인 합계(예: `4,500원`)
- 하단에 총 금액(예: `총 금액 12,500원`)을 표시한다.
- CTA 버튼: `주문하기`

#### 5) 기능 요구사항
##### (1) 메뉴/옵션 데이터 처리
- 프론트는 상품과 옵션 정의를 바탕으로 카드 UI를 렌더링한다.
- 옵션은 가격이 가산되며, `시럽 추가(+0원)`은 총액에는 영향을 주지 않지만 선택 여부는 항목 요약에 반영된다.

##### (2) `담기` 동작(장바구니 아이템 머징)
- 사용자가 특정 메뉴 카드에서 옵션을 선택한 뒤 `담기`를 클릭하면 장바구니에 항목이 추가된다.
- 동일한 상품 + 동일한 옵션 조합이 이미 장바구니에 존재하는 경우:
  - 기존 항목의 수량 `+1`
  - 라인 합계 및 총 금액 재계산
- 동일 옵션 조합이 없으면:
  - 새로운 장바구니 라인아이템 생성
  - 수량은 `x 1`로 시작

##### (3) 장바구니 UI 상태
- 장바구니가 비어 있는 경우:
  - 항목 리스트 영역은 비어 있으며,
  - `주문하기` 버튼은 비활성(`disabled`) 처리한다. (또는 클릭 시 안내)
- 장바구니는 MVP 기준으로 “담기”로만 수량이 증가하는 동작을 제공한다.

##### (4) `주문하기` 동작
- 사용자가 `주문하기` 버튼을 클릭하면 백엔드에 주문 생성 요청을 보낸다.
- 요청에는 최소 다음이 포함되어야 한다.
  - 장바구니 항목 목록
  - 각 항목의 상품 식별자/기본가, 선택 옵션, 수량
- 백엔드 응답이 성공이면:
  - 주문 성공 메시지 표시(토스트/알림 중 택1)
  - 장바구니 초기화
  - (선택) 주문 번호/접수 안내를 표시
- 실패이면:
  - 사용자에게 오류 메시지를 노출
  - 장바구니 상태는 유지

##### (5) 로딩/에러 처리
- `주문하기` 호출 동안 버튼 비활성화 및 로딩 상태를 표시한다.
- 메뉴 로딩 중에는 스켈레톤 또는 로딩 인디케이터를 표시한다.
- 네트워크 오류/서버 오류는 사용자에게 안내한다.

##### (6) 접근성/사용성
- 버튼/체크박스는 키보드 포커스 및 명확한 포커스 표시가 가능해야 한다.
- 가격/총액 및 장바구니의 변경 사항은 사용자에게 시각적으로 즉시 반영되어야 한다.

#### 6) 비기능 요구사항
- 장바구니 조작(`담기`)은 클라이언트 상태 업데이트 기준으로 즉시 반영한다.
- 주문 생성 요청은 `주문하기` 버튼 클릭 시점에만 발생한다. (예: `담기` 시 서버 요청 금지)

#### 7) 수락 기준(Acceptance Criteria)
- A/C 1: `아메리카노(ICE)` + `샷 추가`로 `담기` 시, 장바구니에 `아메리카노(ICE) (샷 추가)` 항목이 생성되고 라인 합계가 `4,500원`으로 표시된다.
- A/C 2: 같은 항목을 다시 `담기` 하면 `x 1`에서 `x 2`로 증가하고, 총 금액이 정확히 재계산된다.
- A/C 3: 장바구니가 비어 있으면 `주문하기` 버튼이 비활성화된다.
- A/C 4: `주문하기` 성공 시 장바구니가 초기화되고 성공 안내가 표시된다.
- A/C 5: `주문하기` 실패 시 오류 안내가 표시되며 장바구니 항목은 유지된다.

### 4.2 관리자 화면

#### 1) 목적
- 관리자가 주문의 처리 상태(주문 접수 → 제조 중 → 제조 완료)를 관리할 수 있도록 한다.
- 관리자가 제품(메뉴)별 재고를 조회하고 증감할 수 있도록 한다.
- 화면 상단 대시보드에서 주문 및 상태 집계를 빠르게 확인할 수 있도록 한다.

#### 2) 사용자
- 관리자: 재고 조정 및 주문 상태 전환을 수행한다.

#### 3) 핵심 사용자 시나리오
1. 관리자가 `관리자` 탭으로 이동한다.
2. 대시보드에서 `총 주문`, `주문 접수`, `제조 중`, `제조 완료` 수치를 확인한다.
3. `재고 현황`에서 특정 메뉴의 재고를 `+/-` 버튼으로 조정한다.
4. `주문 현황`에서 주문 목록을 확인하고, 대기 상태 주문에 대해 `주문 접수` 버튼을 클릭한다.
5. 주문이 처리 진행 상태로 변경되며, 대시보드/주문 목록이 즉시 갱신된다.

#### 4) 화면 구성(UI)
##### (1) 공통 상단 영역
- 앱명 `COZY`
- 상단 탭/네비게이션: `주문하기`, `관리자`
- 현재 화면은 `관리자`가 활성 상태로 표시된다.

##### (2) 관리자 대시보드
- 제목: `관리자 대시보드`
- 지표 4종(와이어프레임 기준):
  - `총 주문 {N}`
  - `주문 접수 {N}`
  - `제조 중 {N}`
  - `제조 완료 {N}`
- 수치는 주문 상태 데이터 기준으로 계산되어 표시된다.

##### (3) 재고 현황
- 제목: `재고 현황`
- 메뉴별 카드(와이어프레임 기준 3개):
  - 아메리카노(ICE)
  - 아메리카노(HOT)
  - 카페라떼
- 각 카드에는 다음 요소가 포함된다.
  - 제품명
  - 현재 재고 수량 `X{재고}`
  - `+` 버튼(재고 증가)
  - `-` 버튼(재고 감소)
- `-`로 재고가 감소할 때 재고가 0 미만으로 내려가지 않도록 한다.

##### (4) 주문 현황
- 제목: `주문 현황`
- 주문 목록(최소 1개 이상)을 리스트 형태로 보여준다.
- 각 주문 아이템은 다음 정보를 포함한다.
  - 주문 일시(예: `7월 31일 13:00`)
  - 주문 내역(예: `아메리카노(ICE) x 1`)
  - 주문 총 금액(예: `4,000원`)
  - 주문 상태에 따른 액션 버튼(와이어프레임 기준):
    - 주문이 `주문 접수` 대기 상태라면 버튼 텍스트 `주문 접수`
    - 주문이 `제조 중` 상태라면(추후 확장) 버튼 텍스트 `제조 완료`를 노출
- 버튼 클릭 시 해당 주문 상태가 변경되고 목록/대시보드가 갱신된다.

#### 5) 기능 요구사항
##### (1) 대시보드 집계
- 프론트는 주문 상태(주문 접수/제조 중/제조 완료) 데이터를 기반으로 대시보드 지표를 계산하거나, 백엔드 제공 값을 렌더링한다.
- `총 주문`은 모든 상태의 주문을 포함한다.

##### (2) 재고 조정
- 관리자는 메뉴별 재고를 `+/-`로 조정할 수 있어야 한다.
- `+` 클릭 시 백엔드에 재고 증가 요청을 보낸다.
- `-` 클릭 시 백엔드에 재고 감소 요청을 보낸다.
- 재고 감소는 0 이하가 되지 않도록 서버에서 검증한다.
- 요청 성공 시 프론트의 `재고 현황` 수량이 즉시 갱신된다.

##### (3) 주문 상태 전환
- 관리자는 `주문 현황`에서 액션 버튼을 통해 주문 상태를 전환한다.
- 주문 상태 전환 규칙(초기 MVP):
  - `주문 접수` 대기 상태 주문 → `주문 접수` 처리 시 `제조 중`으로 전환
- 재고 검증(중요):
  - 주문을 `주문 접수` 처리할 때, 해당 주문의 메뉴 구성에 대해 재고가 충분해야 한다.
  - 재고가 부족하면 상태 전환을 수행하지 않고 오류 메시지를 표시한다.
- 성공 시 해당 주문 상태가 변경되고, 대시보드 지표와 주문 목록이 즉시 갱신된다.

##### (4) 로딩/에러 처리
- 재고 `+/-` 및 주문 상태 전환 버튼 클릭 시 해당 버튼을 비활성화하고 로딩 상태를 표시한다.
- 네트워크 오류/서버 오류 시 오류 안내를 표시하고, UI 상태는 일관되게 유지한다.

#### 6) 비기능 요구사항
- 대시보드 수치와 주문 목록은 상태 전환/재고 조정 성공 후 빠르게 동기화되어야 한다.
- 재고 감소/주문 접수 처리의 유효성 검증(특히 음수 재고, 재고 부족)은 서버에서 보장되어야 한다.

#### 7) 수락 기준(Acceptance Criteria)
- A/C 1: 관리자 진입 시 `관리자 대시보드`는 주문 상태 데이터 기준으로 `총 주문/주문 접수/제조 중/제조 완료` 수치를 정확히 표시한다.
- A/C 2: `재고 현황`에서 `+`를 누르면 해당 메뉴 재고가 1 증가하고 화면에 즉시 반영된다.
- A/C 3: `재고 현황`에서 재고가 0일 때 `-`를 누르면 재고는 0 미만으로 내려가지 않고 오류(또는 무동작)로 처리된다.
- A/C 4: `주문 현황`에서 `주문 접수` 버튼을 클릭했을 때, 재고가 충분하면 주문 상태가 전환되고 대시보드 지표가 변경된다.
- A/C 5: `주문 현황`에서 `주문 접수` 버튼을 클릭했을 때, 재고가 부족하면 주문 상태는 변경되지 않고 오류 안내가 표시된다.

## 5. 백엔드 개발 PRD

백엔드는 **2장 기술 스택**(Node.js, Express, PostgreSQL)을 따르며, 프론트의 주문하기·관리자 화면이 동일한 데이터를 조회·갱신할 수 있도록 API와 스키마를 정의한다. 인증·결제는 범위에서 제외한다.

### 5.1 데이터 모델

엔티티는 최소 다음 세 가지를 포함한다. 실제 테이블명·컬럼명은 구현 시 스네이크 케이스 등 팀 규칙에 맞춰 정하면 된다.

#### (1) 메뉴(Menu / Coffee)

| 항목 | 설명 |
|------|------|
| 식별자 | 메뉴 고유 ID (PK) |
| 이름 | 커피 이름 (예: 아메리카노(ICE)) |
| 설명 | 메뉴 설명 문구 |
| 가격 | 기본 판매 가격(원, 정수) |
| 이미지 | 이미지 URL 또는 저장 경로(프론트에서 카드 썸네일로 사용) |
| 재고 수량 | 현재 보유 수량(정수, 0 이상). **주문하기 화면에는 노출하지 않고**, 관리자·서버 검증용으로 사용한다. |

#### (2) 옵션(Option)

옵션은 특정 메뉴에 종속되거나, 공통 옵션을 메뉴와 **다대다/다대일**로 연결할 수 있다. MVP에서는 **메뉴당 적용 가능한 옵션 목록**을 명시하는 방식이면 충분하다.

| 항목 | 설명 |
|------|------|
| 식별자 | 옵션 고유 ID (PK) |
| 옵션 이름 | 예: 샷 추가, 시럽 추가 |
| 옵션 가격 | 옵션 추가 시 합산되는 금액(원, 0 허용) |
| 연결 메뉴 | 적용 대상 메뉴 ID (FK). 한 옵션이 여러 메뉴에 쓰이면 별도 매핑 테이블 고려 |

#### (3) 주문(Order)

주문 한 건과 그에 속한 **라인 아이템**을 구분한다.

**주문(Order)**

| 항목 | 설명 |
|------|------|
| 식별자 | 주문 고유 ID (PK) |
| 주문 일시 | 주문이 확정된 시각(서버 기준 타임스탬프) |
| 상태 | `주문 접수` → `제조 중` → `제조 완료` 중 하나(열거형) |
| 총 금액 | 라인 합계의 합(검증·표시용, 서버에서 재계산 권장) |

**주문 라인(Order line / 주문 내용)**

한 주문에 여러 메뉴·옵션 조합이 들어갈 수 있으므로 라인 단위로 저장한다.

| 항목 | 설명 |
|------|------|
| 식별자 | 라인 고유 ID (PK) |
| 주문 ID | 소속 주문 (FK) |
| 메뉴 ID | 주문한 메뉴 (FK) |
| 수량 | 해당 메뉴+옵션 조합 수량 |
| 선택 옵션 | 해당 라인에 적용된 옵션(옵션 ID 목록 또는 JSON 등 구현 선택) |
| 라인 금액 | (메뉴 기본가 + 선택 옵션가 합) × 수량 |

> **정리:** “주문 정보”에 포함되는 **주문 일시**는 Order에, **주문 내용(메뉴, 수량, 옵션, 금액)**은 Order line(들)에 담는다.

### 5.2 데이터 스키마 설계를 위한 사용자 흐름

아래 흐름이 만족되도록 DB 제약(외래키, NOT NULL, 재고 비음수 등)과 트랜잭션 경계를 설계한다.

1. **메뉴 조회(주문하기)**  
   서버는 메뉴 목록(이름, 설명, 가격, 이미지 등)을 반환한다. **재고 수량은 주문하기 API 응답에 포함하지 않거나**, 포함하더라도 프론트 주문하기 화면에서는 사용하지 않는다. 재고는 **관리자 화면**에서만 조회·표시한다.

2. **장바구니(클라이언트)**  
   사용자가 메뉴·옵션을 선택해 담으면, 선택 정보는 브라우저의 장바구니 상태로 표시된다(서버 세션 필수 아님).

3. **주문 확정**  
   사용자가 `주문하기`를 누르면, 장바구니 내용을 바탕으로 **주문(Order)과 주문 라인들**을 데이터베이스에 저장한다.  
   동시에 정책에 따라 **해당 메뉴의 재고를 차감**한다(부족 시 저장 실패·롤백).  
   저장되는 정보: 주문 일시, 각 라인의 메뉴·수량·옵션·라인 금액, 주문 총액, 초기 상태.

4. **관리자 주문 목록**  
   저장된 주문을 관리자 화면의 주문 목록에 표시한다.  
   - 신규 주문의 **기본 상태**는 `주문 접수`이다.  
   - 관리자가 동작(버튼)을 수행하면 `주문 접수` → `제조 중` → `제조 완료` 순으로 상태가 변경된다(이전 화면 PRD의 상태 전환·재고 검증과 일치).

### 5.3 API 설계

REST 스타일을 기준으로 하며, 실제 경로·필드명은 구현 시 조정 가능하다. 응답은 JSON을 사용한다.

#### (1) 커피 메뉴 목록 조회

- **목적:** 주문하기 화면 진입 또는 새로고침 시 DB에서 커피 메뉴 목록을 불러온다.
- **예시:** `GET /api/menus`
- **응답:** 메뉴 배열. 각 항목에 이름, 설명, 가격, 이미지 URL, **적용 가능 옵션 목록**(이름·가격) 포함.  
  - 주문하기 전용이라면 **재고 필드는 생략**하거나 `null`로 두어 프론트가 노출하지 않게 한다.

#### (1-1) 관리자용 메뉴/재고 조회

- **목적:** 관리자 화면의 `재고 현황`에서 메뉴별 재고 수량과 상태를 표시한다.
- **예시:** `GET /api/admin/menus`
- **응답:** 메뉴 배열. 각 항목에 이름, 설명, 가격, 이미지 URL, 재고 수량, 재고 상태 계산에 필요한 값을 포함한다.
- **재고 상태 규칙:** 백엔드 또는 프론트 중 한 곳에서 일관되게 계산한다.
  - 재고 `0개`: `품절`
  - 재고 `1~4개`: `주의`
  - 재고 `5개 이상`: `정상`

#### (2) 주문 생성

- **목적:** 사용자가 `주문하기`를 누르면 주문 정보를 DB에 저장하고, 정책에 따라 **메뉴 재고를 수정**한다.
- **예시:** `POST /api/orders`
- **요청 본문:** 라인 배열(메뉴 ID, 수량, 선택한 옵션 ID 목록 등). 서버는 금액·총액을 **재검증**한다.
- **처리:**  
  - 트랜잭션 내에서 재고 확인 → 주문·라인 insert → 메뉴별 재고 차감.  
  - 재고 부족 시 `4xx`와 명확한 메시지, DB 변경 없음.
- **응답:** 생성된 주문 ID, 주문 일시, 총액, 초기 상태 등.

#### (3) 주문 단건 조회

- **목적:** 주문 ID를 알면 해당 주문의 상세 정보를 조회한다.
- **예시:** `GET /api/orders/:orderId`
- **응답:** 주문 메타(일시, 상태, 총액) + 라인 목록(메뉴명·수량·옵션·라인 금액 등 표시용 필드).

#### (4) 관리자·추가 API(화면 PRD와 정합)

다음은 백엔드 구현 시 함께 정의하면 프론트와 맞물리기 쉽다.

- **주문 목록:** `GET /api/orders` (쿼리로 상태 필터, 정렬: 최신순 등)
- **주문 상태 변경:** `PATCH /api/orders/:orderId/status` (허용 전환만 서버에서 검증)
  - `주문 접수` 상태에서 제조 시작 요청 시 `제조 중`으로 변경한다.
  - `제조 중` 상태에서 완료 요청 시 `제조 완료`로 변경한다.
  - 이미 `제조 완료`인 주문은 더 이상 상태를 변경하지 않는다.
- **재고 조정:** `PATCH /api/menus/:menuId/stock` (증감 값 또는 절대값; 0 미만 방지)

에러 시 HTTP 상태 코드와 한글/영문 메시지 본문을 통일해 프론트에서 안내한다.

### 5.4 데이터베이스 스키마 초안

PostgreSQL 기준의 MVP 스키마 초안이다. 실제 컬럼 타입과 제약은 구현 시 조정할 수 있으나, 아래 관계와 제약은 유지한다.

#### (1) `menus`

| 컬럼 | 타입 예시 | 설명 |
|------|-----------|------|
| `id` | `bigserial` | 메뉴 ID |
| `name` | `varchar(100)` | 커피 이름 |
| `description` | `text` | 메뉴 설명 |
| `price` | `integer` | 기본 가격 |
| `image_url` | `text` | 이미지 경로 또는 URL |
| `stock_quantity` | `integer` | 재고 수량, 0 이상 |
| `created_at` | `timestamp` | 생성 일시 |
| `updated_at` | `timestamp` | 수정 일시 |

필수 제약:
- `price >= 0`
- `stock_quantity >= 0`
- `name`은 빈 값이 될 수 없다.

#### (2) `menu_options`

| 컬럼 | 타입 예시 | 설명 |
|------|-----------|------|
| `id` | `bigserial` | 옵션 ID |
| `menu_id` | `bigint` | 연결 메뉴 ID |
| `name` | `varchar(100)` | 옵션 이름 |
| `price` | `integer` | 옵션 가격 |
| `created_at` | `timestamp` | 생성 일시 |
| `updated_at` | `timestamp` | 수정 일시 |

필수 제약:
- `menu_id`는 `menus.id`를 참조한다.
- `price >= 0`
- 같은 메뉴에 동일 옵션명이 중복되지 않도록 한다.

#### (3) `orders`

| 컬럼 | 타입 예시 | 설명 |
|------|-----------|------|
| `id` | `bigserial` | 주문 ID |
| `ordered_at` | `timestamp` | 주문 접수 일시 |
| `status` | `varchar(30)` | `주문 접수`, `제조 중`, `제조 완료` |
| `total_amount` | `integer` | 주문 총 금액 |
| `created_at` | `timestamp` | 생성 일시 |
| `updated_at` | `timestamp` | 수정 일시 |

필수 제약:
- `status`는 허용된 상태값만 저장한다.
- 신규 주문의 기본 상태는 `주문 접수`이다.
- `total_amount >= 0`

#### (4) `order_items`

| 컬럼 | 타입 예시 | 설명 |
|------|-----------|------|
| `id` | `bigserial` | 주문 항목 ID |
| `order_id` | `bigint` | 주문 ID |
| `menu_id` | `bigint` | 메뉴 ID |
| `menu_name` | `varchar(100)` | 주문 당시 메뉴명 스냅샷 |
| `unit_price` | `integer` | 주문 당시 메뉴 기본가 |
| `quantity` | `integer` | 주문 수량 |
| `line_amount` | `integer` | 항목 합계 금액 |

필수 제약:
- `order_id`는 `orders.id`를 참조한다.
- `menu_id`는 `menus.id`를 참조한다.
- `quantity > 0`
- `line_amount >= 0`
- 메뉴명이 추후 수정되어도 과거 주문 표시가 깨지지 않도록 주문 당시 `menu_name`, `unit_price`를 스냅샷으로 저장한다.

#### (5) `order_item_options`

| 컬럼 | 타입 예시 | 설명 |
|------|-----------|------|
| `id` | `bigserial` | 주문 옵션 ID |
| `order_item_id` | `bigint` | 주문 항목 ID |
| `option_id` | `bigint` | 옵션 ID |
| `option_name` | `varchar(100)` | 주문 당시 옵션명 스냅샷 |
| `option_price` | `integer` | 주문 당시 옵션 가격 |

필수 제약:
- `order_item_id`는 `order_items.id`를 참조한다.
- `option_id`는 `menu_options.id`를 참조한다.
- 옵션명이 추후 수정되어도 과거 주문 표시가 유지되도록 `option_name`, `option_price`를 스냅샷으로 저장한다.

### 5.5 백엔드 수락 기준(Acceptance Criteria)

- A/C 1: `GET /api/menus` 호출 시 주문하기 화면에 필요한 메뉴명, 설명, 가격, 이미지, 옵션 목록이 반환된다.
- A/C 2: `GET /api/admin/menus` 호출 시 메뉴별 재고 수량이 반환되고, 관리자는 재고 상태(정상/주의/품절)를 판단할 수 있다.
- A/C 3: `POST /api/orders` 호출 시 주문 일시, 주문 항목(메뉴, 수량, 옵션, 금액), 총 금액, 초기 상태(`주문 접수`)가 DB에 저장된다.
- A/C 4: 주문 생성 시 재고가 충분하면 메뉴 재고가 주문 수량만큼 차감된다.
- A/C 5: 주문 생성 시 재고가 부족하면 주문과 재고 변경이 모두 저장되지 않고 오류가 반환된다.
- A/C 6: `GET /api/orders/:orderId` 호출 시 주문 일시, 상태, 메뉴, 수량, 옵션, 금액이 포함된 주문 상세 정보가 반환된다.
- A/C 7: 관리자 주문 상태 변경 API는 `주문 접수` → `제조 중` → `제조 완료` 순서만 허용한다.
- A/C 8: 재고 조정 API는 재고 수량이 0 미만으로 내려가지 않도록 서버에서 검증한다.
